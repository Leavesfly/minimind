# 16 - 训练方法基本原理：从第一性原理理解每种训练为什么有效

> 从第一性原理出发，解释 MiniMind3 中各训练方法**为什么有效**。不堆公式，用直觉和类比让你真正理解每种方法在做什么。

---

## 全景图：一个模型是如何一步步变聪明的

```
原始语料 ──→ [预训练] ──→ 会说话的模型
                              │
                         [监督微调 SFT] ──→ 会对话的模型
                              │
              ┌───────────┬───┼───┬───────────┬──────────┐
              ↓           ↓   ↓   ↓           ↓          ↓
         [LoRA 适配] [DPO] [PPO] [GRPO]  [Agent RL] [知识蒸馏]
         领域专家化  偏好   探索  轻量RL  工具调用   压缩知识
                     对齐   进化
```

> 注：SFT 之后的各方法是**并行可选**的，互相之间没有强制先后依赖（详见第十章）。

每一步都在上一步的基础上，赋予模型新的能力。下面我们逐一解释每种方法的核心原理。

## 一、预训练（Pretrain）—— 学会语言本身

### 核心问题：如何让一堆随机数字变成能说话的模型？

### 第一性原理

**语言的本质是序列预测。** 人类在读一句话时，大脑会不断预测下一个词。比如"今天天气很__"，你的大脑会自动补出"好"或"热"。预训练就是让模型学会这件事。

### 类比

想象你是一个外星人，被丢到地球上，不懂任何语言。但你有一个超能力：你能看到人类写过的所有文字。你开始做一件简单的事——**每看到一段文字，就猜下一个字是什么**。

- 第一天：你瞎猜，全错
- 第一个月：你发现"的"字出现得特别频繁
- 第一年：你学会了语法结构
- 十年后：你理解了物理、历史、编程……

这就是预训练。**只需要一个目标：猜下一个 token。** 其他一切知识都是这个过程的副产品。

### 为什么有效

1. **任务足够简单**：next-token prediction 只需要一个交叉熵损失函数，不需要人工标注
2. **数据无限廉价**：互联网上的文本就是天然的训练数据，每个 token 都是标签
3. **知识是压缩的副产品**：为了准确预测下一个 token，模型被迫学会语法、事实、推理等一切语言规律。比如为了预测"水在 100°C 时会__"，模型必须"知道"水的沸点

### 在 MiniMind3 中

```
输入：  "今天天气很好，适合出去"
标签：  "天天气很好，适合出去散"
损失：   每个位置的交叉熵之和
```

padding 位置标记为 -100，不参与 loss 计算。就这么简单。

## 二、监督微调（SFT）—— 学会对话的格式

### 核心问题：预训练后的模型会"续写"，但不会"对话"。怎么教它？

### 第一性原理

**格式就是约束。** 预训练模型学会了语言，但它不知道"当人问你问题时，你应该回答"这件事。SFT 的本质是——**教模型认识一种新的格式：问→答**。

### 类比

想象一个博学的教授，他读过所有书籍，但从没和人说过话。你需要给他看一些对话范例：

```
学生：什么是光合作用？
教授：光合作用是植物利用阳光将二氧化碳和水转化为葡萄糖和氧气的过程。
```

看了几千个这样的范例后，教授就学会了"别人问我问题时，我应该给出有条理的回答"。

### 为什么有效

1. **格式本身是简单的模式**：对话的 `user → assistant` 交替模式，模型只需几百到几千条数据就能学会
2. **Label Mask 的巧妙设计**：训练时只对 assistant 的回复计算 loss，不让模型学"如何提问"。这就像老师只看学生的答卷，不看题目
3. **知识不需要重新学**：预训练已有的知识只需要被"激活"为对话形式，而非重新学习

### 关键设计：为什么只算 assistant 部分的 loss？

```
[user: 什么是AI？]  [assistant: AI是人工智能的缩写...]
      -100                    计算 loss
```

如果对 user 部分也计算 loss，模型会学习"如何提问"，这不是我们要的。只标注 assistant 回复为有效 label，模型只学习"如何回答"。

## 三、LoRA —— 用 0.1% 的参数学会新技能

### 核心问题：微调一个大模型成本太高，能不能只调一小部分参数？

### 第一性原理

**高维空间中的变化往往是低秩的。** 一个 512×512 的权重矩阵有 26 万个参数，但微调时真正需要改动的"方向"可能只有几十个。LoRA 就是只学那几十个方向。

### 类比

想象你是一个画家，已经很会画风景画了。现在要学画人物。你不需要重新学怎么拿画笔、怎么调颜色——你只需要学"人体比例"和"表情"这两个新维度。

LoRA 做的就是这件事：冻结整个模型（你已有的绘画技能），只训练两个小矩阵 A 和 B（新维度）：

```
原始权重 W (512×512) → 冻结不动
新增：A (512×16) × B (16×512) = ΔW
最终：W + ΔW
```

16 就是 LoRA 的"rank"——你只学 16 个新方向，但这 16 个方向足够覆盖"身份认知"或"医疗知识"这类新能力。

### 为什么有效

1. **微调是低秩的**：研究表明，大模型微调时权重的变化矩阵确实是低秩的，rank=16 就能捕获 99% 的变化
2. **计算效率极高**：512×512 = 262,144 个参数 vs 512×16 + 16×512 = 16,384 个参数，减少 93%
3. **可插拔**：不同任务训练不同 LoRA adapter，推理时热切换，一个基座服务多个场景

### 在 MiniMind3 中

`lora_identity.jsonl`（22KB，90 条数据）就能让模型知道自己叫 MiniMind。这就是 LoRA 的威力——极少数据 + 极少参数 = 精准能力植入。

## 四、DPO —— 直接告诉模型"哪个更好"

### 核心问题：SFT 后的模型能对话了，但有时回答质量参差不齐。怎么教它分辨好坏？

### 第一性原理

**人类的偏好可以直接优化，不需要中间人。** 传统方法（RLHF）需要先训练一个"奖励模型"当裁判，再用强化学习优化。DPO 说：我们可以跳过裁判，直接从偏好数据学习。

### 类比

想象你在教一个厨师做菜。传统方法是：

1. 先培训一个美食评委（奖励模型）
2. 然后让厨师做菜，评委打分，厨师根据分数改进

DPO 的方法是：

1. 直接给厨师看两道菜："这道 A 比那道 B 好吃"
2. 厨师自己琢磨为什么 A 更好，直接调整自己的做菜方式

没有中间环节，更直接，更稳定。

### 数学直觉（不怕，很简单）

DPO 的核心公式只做一件事：

```
让模型对 "好回答" 的概率 ↑
让模型对 "差回答" 的概率 ↓
同时不要偏离原始模型太远（用 reference model 锚定）
```

具体来说：
```
loss = -log σ(β × (好的改善幅度 - 差的改善幅度))
```

- "改善幅度" = 当前模型概率 - 参考模型概率
- β 控制优化力度：太大容易过拟合，太小学不到东西

### 为什么有效

1. **数学等价**：DPO 在数学上与 RLHF 的最优策略完全等价，但不需要训练奖励模型
2. **训练稳定**：没有 RL 训练中的方差问题、奖励作弊问题
3. **数据高效**：只需要成对的偏好数据（chosen vs rejected），标注成本低

### 在 MiniMind3 中

```
{
  "chosen":   [{"role": "user", "content": "问题"}, {"role": "assistant", "content": "详细、有条理的好回答"}],
  "rejected": [{"role": "user", "content": "问题"}, {"role": "assistant", "content": "敷衍、不准确的差回答"}]
}
```

模型学到的不是"记住好回答"，而是"理解什么样的回答风格是好的"。

## 五、知识蒸馏（Distillation）—— 站在巨人肩膀上

### 核心问题：大模型很聪明但太大了跑不动，小模型跑得快但不够聪明。能不能把大模型的知识"塞进"小模型？

### 第一性原理

**概率分布比标签包含更多信息。** 当大模型说"猫"的概率是 0.7、"狗"是 0.2、"兔"是 0.08 时，这些"暗知识"（dark knowledge）比简单的标签"猫"含有丰富得多的信息。

### 类比

想象一个学霸在考试时不仅写了正确答案，还在草稿纸上写了"为什么排除其他选项"。现在让学渣抄作业：

- **普通抄法（Hard Label）**：只抄答案 → 学渣只知道答案是 A，不知道为什么
- **蒸馏抄法（Soft Label）**：抄整个概率分布 → 学渣知道 A 有 70% 可能，B 有 20%，C 和 D 加起来只有 10%。这暗示了 A 和 B 在某些方面相似，而 C、D 完全不靠谱

### 温度（Temperature）的直觉

```
T=1：[0.9, 0.05, 0.03, 0.02]  → 分布很尖锐，暗知识被"压扁"了
T=2：[0.6, 0.2, 0.12, 0.08]   → 分布变平滑，暗知识暴露出来了
```

温度越高，分布越平滑，小概率事件的信息越多。蒸馏时用 T>1 软化教师的输出，让学生能学到更多"选项之间的关系"。

### 为什么有效

1. **信息密度高**：一个 soft label 向量包含的信息量远大于 one-hot label
2. **正则化效果**：soft label 天然带有平滑效果，防止学生过拟合
3. **梯度更有效**：每个训练样本都能提供关于整个词表的梯度信号，而非只有正确 token 那一个位置

### 在 MiniMind3 中

混合损失 = α × KL(学生 ∥ 教师) + (1-α) × CE(学生, 真实标签)

- KL 部分：跟老师学（学习暗知识）
- CE 部分：跟答案学（保证基本正确性）
- α=0.5：两者各占一半，平衡探索与稳定

## 六、PPO —— 让模型自己探索并学习

### 核心问题：SFT 只能从人写的范例学习，如果模型能自己尝试并从反馈中学习，会不会更好？

### 第一性原理

**学游泳不能只看教程，得下水。** SFT 像是给模型看游泳教学视频，PPO 则是把模型推下水，让它自己扑腾，做得好给糖，做得差挨罚。

更准确地说：**强化学习的本质是"试错 + 信用分配"**。模型生成一个完整的回答（试错），然后根据回答的质量得到奖励，再把这个奖励"归因"到每一步的决策上（信用分配）。

### 类比

想象训练一只小狗：

1. **Rollout（试错）**：小狗尝试做一个动作序列（坐下→举爪→翻滚）
2. **Reward（奖励）**：做完后给它打分（翻滚完美 +3，但过程中摔了 -1）
3. **Advantage（归因）**：哪个动作贡献最大？用 GAE 算法来分配功劳
4. **Update（学习）**：增大"好动作"的概率，减小"坏动作"的概率
5. **Clip（安全带）**：每次改进幅度不能太大，防止"忘记怎么走路"

### PPO 的四个关键组件

| 组件 | 角色 | 直觉 |
|------|------|------|
| **Actor** | 生成回答的策略模型 | 正在学习的运动员 |
| **Critic** | 评估每个状态的价值 | 教练，判断"你现在处境好不好" |
| **Reference Model** | 冻结的 SFT 模型 | 安全绳，防止跑偏太远 |
| **Reward Model** | 对回答质量打分 | 裁判 |

### 为什么需要 Clip？

PPO 的核心创新是 clip 机制：

```
更新幅度 = min(优势 × 概率比率, 优势 × clip(概率比率, 1-ε, 1+ε))
```

直觉：**不管反馈多好，每一步最多只能改这么多。** 就像减肥不能一天减 10 斤，否则身体受不了。

### 为什么有效

1. **探索能力**：模型能发现人类标注数据中没有的、更好的回答方式
2. **奖励塑形**：可以设计多维奖励（长度、格式、内容质量、重复惩罚等），引导模型向多个方向同时优化
3. **GAE 优势估计**：精确地把"回答好"的功劳分配到具体的 token 上，学习效率高
4. **KL 约束**：保证模型不会为了获取奖励而"作弊"（如生成乱码骗分）

### 在 MiniMind3 中的奖励设计

```
总奖励 = 长度奖励(±0.5) + 思考奖励(+1.0/-0.5) + 格式奖励(+0.25) - 重复惩罚 + RM评分
```

多个奖励维度协同工作，让模型学会：写适当长度的回答、先思考再作答、不重复啰嗦。

## 七、GRPO —— 去掉裁判，让选手互相比较

### 核心问题：PPO 需要训练一个 Critic 模型来估计价值，能不能省掉这个开销？

### 第一性原理

**好坏是相对的，不需要绝对标准。** PPO 需要 Critic 来说"这个状态值 3.5 分"，但 GRPO 说：不需要绝对评分，只需要知道"这组回答中哪个比较好"就够了。

### 类比

想象你是写作老师，要评价学生的作文：

- **PPO 方式**：给每篇作文打一个绝对分数（85 分、72 分……），然后告诉学生哪里做得好
- **GRPO 方式**：让同一个学生对同一个题目写 4 篇作文，然后排名——"第 2 篇最好，第 4 篇最差"。学生从排名中学习

GRPO 的优势：**不需要训练一个评分裁判（Critic），直接从组内相对排名学习。**

### 核心算法

```
1. 对同一个 prompt，模型生成 G 个回答（一组）
2. 对每个回答计算奖励
3. 在组内标准化：advantage = (reward - 组平均) / 组标准差
4. 用标准化后的优势更新策略
```

### 为什么"组内比较"就够了？

- 一个绝对分数为 "7 分" 的回答，好不好？如果平均水平是 3 分，它很好；如果平均水平是 9 分，它很差
- GRPO 直接用相对排名，跳过了"绝对分数"这个中间环节
- 数学上，当组足够大时，组内均值就是对 Critic 价值函数的无偏估计

### 为什么有效

1. **省去 Critic**：不需要训练额外的价值网络，节省 50% 的 GPU 显存和计算量
2. **方差更低**：组内标准化天然降低了梯度估计的方差
3. **训练更稳定**：没有 Critic 估计不准导致的误差累积问题
4. **实现简单**：核心逻辑不到 50 行代码

### CISPO 变体

MiniMind3 中还实现了 CISPO（Constrained Importance Sampling Policy Optimization），它在 GRPO 基础上加了一个约束：

```
如果某个 token 的概率比率偏离太远 → 直接 clip 掉
```

相当于给 GRPO 也加了一条安全带，防止单次更新过大。

## 八、Agent RL —— 教模型使用工具

### 核心问题：模型只能"说"，但不能"做"。怎么教它学会调用外部工具？

### 第一性原理

**语言不仅是表达，还是行动。** 当你说"帮我查下明天天气"时，你期望的不是一段关于天气的文学描写，而是一次真实的 API 调用。Agent RL 教模型学会"什么时候该调用工具、调用哪个工具、传什么参数"。

### 类比

想象教一个实习生使用公司内部系统：

1. 给他一本操作手册（tools 定义）
2. 给他一个任务："查一下订单 #12345 的状态"
3. 他尝试操作（可能点错按钮、填错参数）
4. 看看结果对不对（用 ground truth 验证）
5. 做对了奖励，做错了重来

经过多次练习，实习生学会了：
- **什么时候该查系统**（而非自己编答案）
- **用哪个功能**（查订单 vs 查库存）
- **填什么参数**（订单号的格式）

### 与普通 RL 的区别

| 普通 RLAIF | Agent RL |
|-----------|----------|
| 生成一次性回答 | 多轮交互（调用工具→获取结果→继续推理） |
| 奖励基于文本质量 | 奖励基于工具调用正确性 + 最终答案正确性 |
| 单步 rollout | 多步 rollout（可能调用多次工具） |
| 不需要外部环境 | 需要工具执行环境（模拟或真实） |

### 多轮 Rollout 的过程

```
Prompt: "算算 7109 × 2920"
  ↓
模型输出: <tool_call>{"name": "calculate_math", "arguments": {"expression": "7109*2920"}}</tool_call>
  ↓
环境执行: 返回 "20758280"
  ↓
模型继续生成: "7109 × 2920 = 20758280"
  ↓
验证 gt: ["20758280"] ✓ → 奖励 +1
```

### 为什么有效

1. **格式化行为通过 RL 自然涌现**：模型不是被强制记忆工具调用格式，而是通过试错学会——用对了格式才能得到工具返回值，才能答对
2. **泛化能力强**：不是死记"7109×2920=20758280"，而是学会"遇到计算题就调计算器"这个策略
3. **Ground Truth 验证闭环**：`gt` 字段提供客观的正确答案，奖励信号无歧义
4. **多轮交互学习计划性**：模型学会"先分析问题→调用合适工具→整合结果→回答"的完整流程

### 在 MiniMind3 中

Agent RL 的奖励包括多个维度：
- **格式正确性**：工具调用的 JSON 格式是否合法
- **工具选择正确性**：是否调用了正确的工具
- **参数正确性**：传入的参数是否有效
- **最终答案正确性**：与 `gt` 对比的结果

## 九、各方法对比总结

### 它们解决的是什么问题？

| 方法 | 解决的核心问题 | 一句话总结 |
|------|--------------|-----------|
| **预训练** | 从零学习语言 | 通过预测下一个 token 压缩世界知识 |
| **SFT** | 学会对话格式 | 看范例学习"问→答"的模式 |
| **LoRA** | 低成本领域适配 | 冻结大部分参数，只学几个新方向 |
| **DPO** | 价值观对齐 | 直接从"好 vs 差"的对比中学习偏好 |
| **蒸馏** | 模型压缩 | 从大模型的概率分布中提取暗知识 |
| **PPO** | 自主探索进化 | 试错 + 奖励反馈 + Actor-Critic |
| **GRPO** | 高效策略优化 | 去掉 Critic，用组内相对排名替代 |
| **Agent RL** | 学会使用工具 | 多轮交互中学习"何时用什么工具" |

### 信息来源从哪里？

```
预训练：信息来自原始文本（无标注）
SFT：  信息来自人类写的范例对话
LoRA： 信息来自少量领域数据
DPO：  信息来自人类偏好判断（A比B好）
蒸馏： 信息来自大模型的概率分布
PPO：  信息来自奖励模型的评分
GRPO： 信息来自组内相对比较
Agent：信息来自工具执行的真实反馈
```

### 演化路径

```
不会说话 ──[预训练]──→ 会续写
         ──[SFT]────→ 会对话
         ──[DPO]────→ 回答质量高
         ──[PPO]────→ 自主发现更好的表达
         ──[Agent]──→ 会使用工具解决问题
```

每一步都是在上一步能力的基础上，增加一个新的维度。这就是现代 LLM 训练的完整故事。

## 十、训练顺序：必须有先后吗？为什么？

### 短答案

**预训练 → SFT 是严格必须的先后顺序。** SFT 之后的阶段（DPO、RL、LoRA、蒸馏）则是**并行可选**的，互相之间没有强制依赖。

### 完整的依赖关系图

```
预训练（必须第一步）
  │
  ↓ 严格依赖
SFT（必须第二步）
  │
  ├──→ DPO（可选）
  ├──→ PPO / GRPO（可选）
  ├──→ Agent RL（可选）
  ├──→ LoRA（可选）
  └──→ 蒸馏（可选，需要一个已有的大模型当老师）
```

### 为什么预训练必须在最前面？

**类比：你不能教一个不识字的人写作文。**

预训练让模型学会"语言是什么"——词汇、语法、事实知识、推理模式。如果跳过预训练直接做 SFT：

- 模型的权重是随机初始化的
- 它连"的"字是什么都不知道
- 给它看再多对话范例，它也只是在胡乱拟合，不会泛化

**信息论角度**：预训练的本质是在模型参数中压缩整个语言的统计规律。这是后续一切能力的地基。没有地基，上面什么都建不起来。

### 为什么 SFT 必须在预训练之后？

**类比：你不能教一个只会说方言的孩子直接写英文论文，得先教他英文。**

预训练后的模型只会"续写"——你给它一段话，它接着写下去。但它不理解"对话"这个格式：

```
预训练模型看到 "用户：你好" → 它可能续写 "用户：你也好" （模仿了格式继续编）
SFT 后的模型看到 "用户：你好" → "助手：您好，有什么可以帮助您的？" （理解了角色）
```

SFT 的作用是在预训练建立的语言能力之上，**注入对话格式的理解**。它需要预训练提供的语言基础，但预训练不需要 SFT。

### 为什么 DPO/RL 不能在 SFT 之前做？

**类比：你不能对一个还不会做饭的人说"你做的 A 菜比 B 菜好吃"——他两道都做不出来。**

- **DPO** 需要模型能生成合理的回答，才能比较"哪个更好"。如果模型还不会对话，chosen 和 rejected 对它来说都是噪音
- **PPO/GRPO** 需要模型能生成有意义的 rollout。如果模型连一句完整的话都说不出来，奖励信号就毫无意义
- **Agent RL** 需要模型至少能生成工具调用的格式片段，否则探索空间无穷大，永远学不到有效策略

**核心原理**：RL 需要一个"足够好的起点"来探索。SFT 提供的就是这个起点——一个已经知道怎么对话的模型。

### SFT 之后的各方法，互相有先后吗？

**大部分没有强制先后顺序**，但有推荐顺序和实践经验：

| 组合 | 是否必须有先后 | 原因 |
|------|--------------|------|
| SFT → DPO | **推荐但非必须** | DPO 假设模型已有基本对话能力。理论上可以合并到 SFT 同时训练，但分开更稳定 |
| SFT → PPO/GRPO | **推荐但非必须** | RL 需要模型能生成有意义的文本作为探索的起点 |
| DPO → RL | **不需要** | 它们解决不同问题，可以独立进行。实践中通常选其一 |
| SFT → LoRA | **推荐** | LoRA 通常是在 SFT 后的模型上做领域适配。但技术上也可以直接对预训练模型做 LoRA |
| SFT → 蒸馏 | **不需要** | 蒸馏可以在预训练阶段就做（预训练蒸馏），也可以在 SFT 阶段做 |
| PPO → Agent RL | **不需要** | Agent RL 是独立的 RL 训练流程，直接从 SFT 模型开始即可 |

### 一个常见误解

> "是不是必须 预训练 → SFT → DPO → PPO → Agent RL 全走一遍？"

**不是。** 正确的理解是：

```
必须：预训练 → SFT（这两步是地基）

可选菜单（根据需要选择一个或多个）：
  □ DPO    —— 如果你想对齐偏好
  □ PPO    —— 如果你想让模型自主进化
  □ GRPO   —— 如果你想要更轻量的 RL
  □ Agent  —— 如果你想让模型会用工具
  □ LoRA   —— 如果你想低成本适配新领域
  □ 蒸馏   —— 如果你想压缩模型
```

实际项目中，通常只会选其中 1-2 个后续方法。比如 ChatGPT 的路径是 `预训练 → SFT → RLHF(PPO)`，DeepSeek-R1 的路径是 `预训练 → SFT → GRPO`。

### 为什么是这样的顺序？——从信息论角度的统一解释

每一步都在向模型注入一种新类型的信息，而这种信息的注入**需要前一步的信息作为基础**：

```
第1步 预训练：注入"语言的统计规律"
  → 需要：无（从零开始）
  → 产出：理解语言的模型

第2步 SFT：注入"对话的格式规范"  
  → 需要：理解语言（否则格式学不会）
  → 产出：会对话的模型

第3步+ 对齐/RL：注入"什么是好回答"
  → 需要：会对话（否则没有可以比较/改进的基础）
  → 产出：回答质量高的模型
```

**本质是一个从粗到细的学习过程**：先学大的模式（语言），再学小的模式（对话格式），最后打磨细节（质量、偏好、工具使用）。就像学画画：先学素描（结构），再学上色（表达），最后学风格（个性）。如果顺序反了，就像先教一个人"梵高的笔触风格"，但他连一条直线都画不了。

## 十一、延伸思考

### 为什么不能只用 RL 从零训练？

RL 的探索空间太大了。如果从随机策略开始，模型连一个有意义的句子都生成不出来，更别提从奖励中学习。预训练和 SFT 为 RL 提供了一个**好的起点**——模型至少知道怎么说话了，RL 才能在此基础上探索"怎么说得更好"。

### 为什么 SFT 后还需要 RL？

SFT 的天花板是人类标注数据的质量。如果标注者自己都写不出完美回答，模型也学不到更好的。RL 打破了这个天花板——模型可以通过探索发现标注数据中不存在的、更优的回答方式。这就是为什么 ChatGPT 比它的 SFT 版本强得多。

### 这些方法的未来趋势

1. **RL 占比越来越大**：从 GPT-4 到 o1 到 o3，RL 训练的比例在持续增加
2. **Agent 能力是终极目标**：最终我们希望模型不仅能说，还能用工具做事
3. **自我改进**：模型用自己生成的数据训练自己，减少对人类标注的依赖
